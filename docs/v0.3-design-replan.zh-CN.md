## 2. 设计取舍

### 2.1 image2 应采用的方向

image2 是本地生图工作台，不是营销页，也不是暗色数据大屏。最终设计应采用：

```txt
Clean + Professional + Dashboard information hierarchy
```

解释：

- `clean`：提供简洁、低噪声、留白和可读性。
- `professional`：提供可信任、可长期使用的业务工具气质。
- `dashboard`：提供多区域工作台的信息组织能力。

### 2.2 不直接采用的部分

不要照搬以下风格：

- `application` 的紫色主调和玻璃面板。原因：会把工具做成通用 SaaS 外观，且与当前“避免紫蓝渐变”的要求冲突。
- `dashboard` 的深色主题。原因：image2 是高频参数输入和图片查看工具，浅色更适合长期阅读和表单操作。
- `professional` 的大面积黄色主色。原因：黄色适合强调，不适合作为主操作色。
- `refined` 的 serif/display 字体倾向。原因：中文工具优先使用系统中文字体，serif 风格不适合参数密集型工作台。
- 任何 glassmorphism、neumorphism、过度阴影、过度圆角、渐变背景。

## 3. v0.3 设计目标

### 3.1 产品目标

把 image2 做成一个中文本地生图工作台：

- 打开后直接能工作。
- 左侧知道接口和模型状态。
- 中间专注生成参数。
- 右侧查看结果和历史。
- 所有状态、错误、按钮都是中文。
- 视觉层级清楚，不需要读说明才能操作。

### 3.2 设计目标

- 降低卡片堆叠感。
- 提高表单可扫描性。
- 强化主操作“开始生成”。
- 让模型、结果、历史更像生产工具，而不是 demo 列表。
- 移动端保留可用性，但不追求桌面同等密度。

### 3.3 非目标

- 不新增生图功能。
- 不新增供应商能力。
- 不做暗色主题。
- 不做主题切换。
- 不做完整 i18n。
- 不做设计系统 npm 包。


保留 Phase 12 的三栏布局：

```txt

```

Phase 13 不应重新推翻信息架构，只优化视觉和组件层级。

## 5. 组件规范

### 5.1 面板

面板用于主要区域，不要每个小元素都套卡片。

推荐：

```txt
白色背景
1px 边框
6-8px 圆角
轻微阴影或无阴影
紧凑内边距
```

避免：

```txt
大阴影
毛玻璃
紫蓝渐变边框
嵌套卡片
过度圆角
```

### 5.2 按钮

按钮分三类：

```txt
主按钮：开始生成、保存接口源
次按钮：测试连接、刷新模型、复用参数
危险按钮：删除、清空历史
```

要求：

- 主按钮使用蓝色实底。
- 次按钮使用白底边框。
- 危险按钮使用红色文本或红色浅底，不和普通次按钮混淆。
- 禁用按钮降低对比，但仍保持可读。

### 5.3 表单

要求：

- 标签短而明确。
- 输入框高度固定，避免布局跳动。
- textarea 只在提示词区域使用。
- 高级 JSON 配置弱化显示，避免抢主流程。
- 聚焦状态用蓝色描边，不使用发光效果。

### 5.4 状态提示

状态提示分四类：

```txt
info：普通说明
success：成功
warning：需要注意
error：错误
```

错误提示结构：

```txt
主提示：用户能理解的中文
详情：技术信息，可选
```

### 5.5 模型列表

模型列表应具备扫描性：

- 模型名优先。
- 模型 ID 次级显示。
- capability 用小标签。
- 当前选中项有明确边框和浅蓝底。
- 不要把每个模型做成强按钮样式。

### 5.6 结果画廊

结果区优先展示图片。

要求：

- 图片区域固定比例，避免加载后跳动。
- 操作按钮在图片下方统一排列。
- 没有结果时显示明确空状态。

### 5.7 历史记录

历史项应支持快速判断：

- 缩略图
- 模式
- 模型
- 时间
- 提示词摘要
- 主要参数
- 操作

历史项视觉上要比当前结果弱一级，避免和新生成结果抢焦点。

## 6. 新 Phase 13 拆分

### Phase 13A：设计令牌与基础样式

目标：

- 引入 CSS variables。
- 更新字体、颜色、背景、基础控件。
- 不动 JSX 结构。

验收：

- `npm run check` 通过。
- 页面主色不再是当前灰青色。
- 主按钮、次按钮、危险按钮有清晰区分。

提交：

```txt
feat: add visual design tokens
```

### Phase 13B：工作台区域视觉整理

目标：

- 优化左中右三栏面板。
- 优化接口源、模型列表、生成表单、结果、历史的层级。
- 不改后端，不新增功能。

验收：

- 1280px 下三栏清晰。
- 1024px 下输出区换行合理。
- 信息区域不互相抢焦点。

提交：

```txt
feat: refine workbench panels
```

### Phase 13C：移动端与文本溢出修复

目标：

- 修复 390px 宽度下的标签、按钮、历史项、模型名溢出。
- 保证中文长词、URL、模型 ID 能换行。

验收：

- 390px 无横向滚动。
- 四个移动标签可点击且不挤压。
- 按钮文字不溢出。

提交：

```txt
fix: harden responsive workbench layout
```

### Phase 13D：浏览器验证与发布前修补

目标：

- 使用浏览器验证真实页面。
- 检查常见状态：
  - 无接口源
  - 已保存接口源
  - 无模型
  - 有模型
  - 生成结果
  - 历史记录
  - 错误提示

验收：

- `npm run check` 通过。
- 浏览器检查 1280px、1024px、390px。
- 没有明显重叠、溢出、英文残留或乱码。

提交：

```txt
test: verify visual refresh
```

## 9. 质量门槛

每一轮必须满足：

- `npm run check` 通过。
- 不新增后端行为。
- 不破坏 provider、模型发现、文生图、图生图、历史记录。
- 不出现明显英文残留。
- 不出现文字重叠。
- 不出现横向滚动。
- 不把 UI 做成营销页。

## 10. 结论

v0.3 的正确方向不是“换一套皮肤”，而是建立一个适合中文生图工作台的产品级视觉系统。已安装的 design skills 应作为参考库，而不是逐字照抄。image2 的最终风格应是：

```txt
浅色、中文、专业、紧凑、清晰、可长期使用。
```
