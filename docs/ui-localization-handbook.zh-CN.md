# image2 中文化与 UI 重做手册

> 目标：把 image2 从“功能可用的英文工程界面”升级为“中文用户能长期使用的本地生图工作台”。本手册用于指导后续 Codex agents 或人工开发，不直接扩展生图能力。

## 1. 为什么现在是英文且不好看

当前 v0.2 的主要目标是打通功能链路：provider 配置、模型发现、文生图、图生图、历史记录、安全审查。前端文案和 UI 视觉没有被单独作为产品任务验收，因此出现了三个问题：

- 文案硬编码为英文，例如 `Provider`、`Text to image`、`Image to image`、`Connected`、`Test failed`。
- 页面结构偏“技术表单堆叠”，没有围绕真实生图流程组织信息。
- 视觉风格偏默认工程 UI：大量白色卡片、重复边框、灰蓝背景、按钮层级单一，缺少清晰的主操作路径。

v0.3 应当优先解决“中文可用性”和“工作台质感”，而不是继续堆新功能。

## 2. v0.3 产品定位

### 2.1 一句话定位

image2 是一个本地优先的中文 AI 生图工作台：用户填入自己的接口地址和密钥，选择模型后完成文生图、图生图，并保存本地历史。

### 2.2 目标用户

- 有多个生图 API 服务的个人用户。
- 需要在不同模型之间切换测试效果的创作者。
- 想把 API Key 留在本地，不想用第三方在线工作台的用户。
- 需要复用历史参数、对比模型结果的调试型用户。

### 2.3 非目标

- 不做营销落地页。
- 不做账号体系。
- 不做公共云部署界面。
- 不做复杂节点式工作流。
- 不做炫技型视觉大屏。

## 3. 设计方向

### 3.1 推荐方向：安静专业的本地工作台

关键词：

- 中文优先
- 信息密度适中
- 操作路径明确
- 参数可扫描
- 结果可对比
- 状态可恢复
- 安全感强

视觉上应接近“专业工具”，而不是“AI 官网首页”。避免大标题 hero、浮夸渐变、装饰性插画和过度圆角卡片。

### 3.2 设计禁区

- 不要做 landing page，首屏就是工具。
- 不要用大片紫蓝渐变当主视觉。
- 不要把所有区域都做成漂浮卡片。
- 不要用英文术语作为默认显示。
- 不要让用户先读说明才能开始使用。
- 不要在页面里塞大段解释文字。
- 不要把错误信息原样暴露为技术栈堆栈。

## 4. 信息架构

### 4.1 桌面端布局

推荐三栏工作台：

```txt
┌──────────────────────────────────────────────────────────────┐
│ 顶栏：image2 / 当前接口源 / 模型状态 / 设置入口              │
├───────────────┬──────────────────────────┬───────────────────┤
│ 左栏           │ 中央工作区                │ 右栏              │
│ 接口源          │ 文生图 / 图生图表单        │ 结果预览           │
│ 模型选择        │ 参数控制                  │ 历史记录           │
│ 安全状态        │ 生成按钮                  │ 复用/下载/复制      │
└───────────────┴──────────────────────────┴───────────────────┘
```

左栏负责“用哪个接口和模型”，中央负责“生成什么”，右栏负责“结果和历史”。

### 4.2 移动端布局

移动端改为顶部标签：

```txt
接口源 | 生成 | 结果 | 历史
```

默认打开“生成”，当没有接口源时自动引导到“接口源”。

### 4.3 首页默认状态

用户第一次打开时，不要展示空白大表单。应展示一个明确的空状态：

```txt
先添加一个接口源
填写你的 API 地址和密钥，image2 会在本地服务端代你请求模型和生成图片。
```

主按钮：

```txt
添加接口源
```

次按钮：

```txt
查看本地安全说明
```

## 6. 文案规则

### 6.1 按用户语言说话

避免：

```txt
Request failed with HTTP 401.
```

改为：

```txt
连接失败：API 密钥无效或没有权限。
```

必要时在次级详情里保留技术信息：

```txt
详情：服务返回 HTTP 401。
```

### 6.2 按动作写按钮

使用：

- 添加接口源
- 测试连接
- 保存接口源
- 刷新模型
- 开始生成
- 复用参数

避免：

- Submit
- Confirm
- Execute
- Save

### 6.3 安全文案要短而明确

推荐：

```txt
API 密钥只发送给本地服务端，不会写入浏览器历史。
```

不要写成大段安全白皮书。详细内容放文档或帮助弹窗。

## 7. 关键流程

### 7.1 添加接口源

流程：

1. 用户点击“添加接口源”。
2. 输入接口名称、接口地址、API 密钥。
3. 选择接口类型：自动识别 / OpenAI 兼容 / image2 兼容。
4. 点击“测试连接”。
5. 测试成功后点击“保存接口源”。
6. 自动刷新模型。

验收：

- API Key 输入框默认密码模式。
- 已保存接口源只显示密钥尾号或“已保存密钥”。
- 编辑接口地址或密钥后，连接状态变为“需要重新测试”。

### 7.2 模型发现

流程：

1. 选择接口源后自动加载模型。
2. 模型按能力分组：支持文生图、支持图生图、能力未知。
3. 当前模式只展示可用模型。
4. 用户可以手动刷新。

验收：

- 没有模型时给出具体下一步：检查接口类型、刷新模型、添加能力补充。
- 能力未知不应静默消失，应给用户修正入口。

### 7.3 文生图

表单层级：

1. 正向提示词
2. 反向提示词
3. 模型
4. 比例 / 清晰度 / 数量
5. 种子
6. 开始生成

验收：

- 生成按钮在缺少模型或提示词时禁用。
- 生成失败后保留用户输入。
- 生成中不允许重复提交。

### 7.4 图生图

表单层级：

1. 上传参考图
2. 正向提示词
3. 重绘强度
4. 模型
5. 比例 / 清晰度 / 数量
6. 开始生成

验收：

- 上传后显示缩略图、文件名、大小。
- 非 PNG/JPEG/WebP 文件给中文错误。
- 超过 5 MB 给中文错误。

### 7.5 历史记录

历史记录应服务于“复用”和“对比”，不是只做列表。

每条历史展示：

- 缩略图
- 模式：文生图 / 图生图
- 模型名称
- 生成时间
- 主要提示词前 40 字
- 操作：复用、下载、复制、删除

验收：

- 复用文生图历史时恢复全部参数。
- 复用图生图历史时恢复参数，但提示用户重新上传参考图。
- 清空历史必须二次确认。

## 8. 视觉规范

### 8.1 色彩

建议不要沿用单一灰蓝/青色。使用中性色为底，少量功能色建立层级。

推荐色板：

```txt
背景        #F6F7F8
面板        #FFFFFF
主文字      #17202A
次文字      #5F6B76
边框        #D8DEE4
主操作      #2563EB
主操作悬停  #1D4ED8
成功        #16803C
警告        #B7791F
错误        #C2410C
选中底色    #EAF1FF
代码底色    #F1F3F5
```

### 8.2 字体

使用系统中文字体栈：

```css
font-family:
  "Microsoft YaHei UI",
  "Microsoft YaHei",
  "PingFang SC",
  "Noto Sans CJK SC",
  system-ui,
  sans-serif;
```

不要用全大写英文 eyebrow 作为主要层级。

### 8.3 间距

推荐：

```txt
页面边距：24px / 32px
栏间距：16px / 20px
表单项间距：14px
按钮高度：36px 或 40px
面板圆角：6px 或 8px
```

### 8.4 按钮层级

主按钮：

- 开始生成
- 保存接口源

次按钮：

- 测试连接
- 刷新模型
- 复用参数

危险按钮：

- 删除
- 清空历史

不要所有按钮都用同一种高饱和背景。

## 9. 组件重构建议

当前 `App.tsx` 承担过多职责。v0.3 应拆分：

```txt
apps/web/src/
├── components/
│   ├── AppShell.tsx
│   ├── Button.tsx
│   ├── Field.tsx
│   ├── EmptyState.tsx
│   ├── Notice.tsx
│   └── SegmentedControl.tsx
├── features/
│   ├── providers/
│   │   ├── ProviderPanel.tsx
│   │   └── provider-copy.ts
│   ├── models/
│   │   ├── ModelPicker.tsx
│   │   └── model-copy.ts
│   ├── generation/
│   │   ├── GenerationWorkbench.tsx
│   │   ├── TextToImageForm.tsx
│   │   ├── ImageToImageForm.tsx
│   │   └── generation-copy.ts
│   └── history/
│       ├── HistoryPanel.tsx
│       └── history-copy.ts
└── copy/
    └── zh-CN.ts
```

第一版不必做完整 i18n 框架，但所有中文文案应集中到 `copy/zh-CN.ts`，避免继续散落硬编码。

## 10. v0.3 建议任务拆分

### Phase 11：中文化与文案集中

目标：

- 默认 UI 全中文。
- 所有主要文案集中管理。
- 错误提示中文化。

验收：

- 页面上不再出现面向用户的英文按钮和状态。
- 测试更新为中文语义。
- 技术字段仅在高级配置或 JSON 编辑区出现。

提交：

```txt
feat: localize app copy to zh-CN
```

### Phase 12：信息架构重排

目标：

- 重排为“左接口/模型，中生成，右结果/历史”的工作台。
- 移动端使用标签式布局。

验收：

- 首屏能看出下一步该做什么。
- 没有接口源、没有模型、没有历史时都有中文空状态。
- 文生图和图生图切换不会造成表单跳动。

提交：

```txt
feat: redesign workbench layout
```

### Phase 13：视觉系统更新

目标：

- 建立按钮、输入框、面板、状态、历史项的统一样式。
- 改掉默认工程感。

验收：

- 主操作突出，次操作克制，危险操作明确。
- 颜色不再是一套灰蓝色重复。
- 桌面端和移动端无明显重叠或溢出。

提交：

```txt
feat: refresh visual system
```

### Phase 14：可用性与浏览器验证

目标：

- 用真实浏览器检查关键流程。
- 修复响应式、文字溢出、按钮状态、空状态问题。

验收：

- 1280px、1024px、390px 宽度都可用。
- provider 添加、模型刷新、文生图表单、图生图上传、历史复用均能操作。
- `npm run check` 通过。

提交：

```txt
test: verify localized workbench flows
```

### Phase 15：v0.3 发布准备

目标：

- 更新 README 和 release notes。
- 打 `v0.3.0`。

验收：

- README 默认中文。
- 英文文档可作为补充，不作为默认入口。
- 发布说明解释 v0.3 是中文化和 UI 重做版本。

提交：

```txt
chore: prepare v0.3.0 release
```

## 12. 发布建议

这次改动应作为 `v0.3.0`，不是 `v0.2.1`。

原因：

- 默认语言变化会显著影响用户体验。
- UI 信息架构会改变主要使用路径。
- 这不是小修小补，而是产品体验升级。

发布标题建议：

```txt
image2 v0.3.0 中文工作台
```
